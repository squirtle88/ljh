### 第二章：系统分析（3分钟，核心章节）

#### 2.1 数据对象设计

**讲法**：先展示 11 个数据对象的总表，然后挑 3 个最关键的细讲。

**必讲的三组对照**：

| 业务问题 | 我们用的数据结构 | 为什么好 |
|---------|----------------|---------|
| 60个车位怎么存？ | **三维数组** spots[3][4][5] | 用 `spots[f][r][c]` 直接访问任意车位，O(1) |
| 空闲车位怎么管理？ | **顺序栈** FreeSpotStack | push/pop 都是 O(1)，不需要遍历找空位 |
| 车牌怎么快速查位置？ | **哈希表** HashMap（拉链法） | 10007个桶，车牌算哈希直接定位，O(1)平均 |
| 满了排队怎么办？ | **链式队列** WaitingQueue | 入队O(1)，出队O(1)，先到先得 |
| 出场记录怎么存？ | **动态数组** history[] | 满了自动扩容1.5倍，顺序遍历统计方便 |

**关键金句**：
> "车位分配如果用遍历方式，每次入场都要扫 60 个车位找空位，O(n)。我们用一个栈，把空闲车位全部压进去，分配的时候直接 pop 栈顶，O(1)。这就是数据结构的威力。"

#### 2.2 数据对象之间的关系

用一张关系图讲（报告中应该有）：

```
ParkingSystem（全局状态）
  ├─ spots[3][4][5]  ← 车位数组
  ├─ free_spots       ← 空闲车位栈
  ├─ wait_queue       ← 等待队列
  ├─ plate_index      ← 哈希索引
  ├─ history[]        ← 交易记录
  ├─ fee_rule         ← 计费规则
  └─ total_revenue    ← 累计收入
```

**金句**：这些结构各司其职，栈管分配，队管排队，哈希管查询，数组管统计。

### 第三章：系统设计（5分钟，重点演示）

#### 3.1 UI 界面

- 展示主菜单截图/运行效果（12个功能选项 + ASCII 边框）
- 强调是 CLI 纯文本，跨平台兼容

#### 3.2 四大核心功能的算法流程

**每个功能讲 1 分钟，按"输入→处理→输出"三段式讲：**

**① 入场流程**（最核心，必须讲清楚）

```
输入车牌 → 校验(1-15字符) → 检查重复(哈希表) → 判断空位(栈判空)
  ├─ 有空位：stack_pop 获取车位 → 创建 VehicleRecord → ht_insert 索引 → 成功
  └─ 没空位：queue_enqueue 加入排队 → 返回"已排队"
```

**金句**：整个入场过程，哈希查重 O(1)，栈判空 O(1)，栈分配 O(1)，全程不需要遍历。

**② 出场+自动调度**

```
输入车牌 → ht_find 查找 → 计算时长 → calculate_fee 计费 → 生成交易记录
→ 释放车位 → 检查队列
  ├─ 队列有车：queue_dequeue 取出 → 直接分配该车位 → 队头车自动入场
  └─ 队列无车：stack_push 车位回栈
```

**金句**：这个设计最精巧的地方是：车位释放后不是直接回栈，而是先看队列有没有人等。有人等就直接给他，省了一步 push+pop。

**③ 费用计算**

```
规则：免费15分钟 → 起步价5元 → 4元/小时 → 日封顶40元
时长扣免费后 → ceil(÷60) 向上取整到小时 → 起步价 + 小时×费率
```

**举例讲解**（这个举例最能让人理解）：
- 停了 16 分钟：扣 15 分钟免费，剩 1 分钟 → ceil(1/60)=1小时 → 5+1×4 = **9元**
- 停了 75 分钟：扣 15 分钟免费，剩 60 分钟 → ceil(60/60)=1小时 → 5+1×4 = **9元**
- 停了 76 分钟：扣 15 分钟免费，剩 61 分钟 → ceil(61/60)=2小时 → 5+2×4 = **13元**

**④ 查询流程**

```
输入车牌 → ht_find(O(1)查哈希表) → 在场 → 返回位置+入场时间
                                  → 不在场 → find_in_queue(O(n)遍历) → 返回排队位置
```

#### 3.3 数据结构实现亮点（可选，有加分）

- **栈的自动扩容**：初始容量60个，满了 realloc 到120个，永远不会溢出
- **哈希表冲突解决**：拉链法，头插法插入，平均查找长度 < 2
- **动态数组扩容**：初始100条，满了1.5倍扩容，摊销O(1)

### 第五章：总结（1分钟）

**三个要点**：

1. **数据结构选型决定了系统性能**：栈保证 O(1) 分配，哈希保证 O(1) 查询
2. **模块分层让协作高效**：pzx 改底层不影响 yjk，yjk 改业务不影响 zxc
3. **C 语言从零实现所有数据结构**：没有用 STL，全部手写，加深了对底层理解

## 五、可能的提问及回答

### Q1：为什么用栈而不是队列来管理空闲车位？

> 用栈（LIFO）和用队列（FIFO）对空闲车位管理来说，时间复杂度都是 O(1)。我们选栈是因为顺序栈的代码更简单（就是数组+指针），而且车位之间没有优先级区别，LIFO 和 FIFO 在这里效果一样，选实现更简单的。

### Q2：哈希表为什么是 10007 个桶？

> 10007 是一个质数。哈希函数是 `hash * 31 + c`，用质数取模可以让分布更均匀，减少冲突。这是从 Java 的 String.hashCode 借鉴的做法。

### Q3：如果有 1000 辆车同时查，哈希表会变慢吗？

> 哈希表的大小固定为 10007，停车场只有 60 个车位，所以哈希表中最多 60 条记录。60 条分布在 10007 个桶里，平均每个桶不到 1 条记录，冲突极少，查找始终接近 O(1)。

### Q4：为什么不直接用文件系统而用二进制文件？

> 二进制文件读写更快，文件更小。而且我们固定了每条记录的字节布局（文件头→车位→在场记录→队列→交易），加载时按相同顺序读取，不需要解析文本格式。

### Q5：三个人的代码怎么合并的？

> 通过头文件定义接口。pzx 在 ds.h 中声明了所有函数，yjk 的 parking.h include ds.h，zxc 的 ui.h include parking.h。编译器在链接阶段自动把三个人的 .o 文件合并成一个可执行文件。只要接口不变，三个人互不影响。
