# 停车场管理系统 — 存储方案分析

> 基于：停车场管理系统_三人分工方案.md | 课程：数据结构实践 | 语言：C 语言

## 一、总体架构

该系统的存储分为两大层：**运行时内存存储**（6 种数据结构）和**持久化磁盘存储**（二进制文件）。两层之间通过 `file_io.c` 中的 `save_data` / `load_data` 桥接。

---

## 二、运行时存储（6 种内存数据结构）

### 2.1 三维车位数组 — `ParkingSpot spots[3][4][5]`

| 属性 | 说明 |
|------|------|
| 类型 | **静态三维数组**（栈上分配，嵌入 `ParkingSystem` 结构体） |
| 容量 | 3层 × 4排 × 5列 = **60 个车位** |
| 存储内容 | 每个元素包含坐标 `SpotLocation`、状态枚举 `SpotStatus`、车牌号 `char[16]` |
| 访问方式 | 直接索引 `spots[f][r][c]`，O(1) |
| 用途 | 存储所有车位的"当前快照"，是车位状态图的唯一数据源 |

**评价**：固定大小、嵌在顶层结构体中，随 `ParkingSystem` 一起分配/释放，零额外内存管理开销。适合 "60 个车位" 这种已知固定规模。

### 2.2 空闲车位栈 — `FreeSpotStack`

| 属性 | 说明 |
|------|------|
| 类型 | **动态数组**（`SpotLocation *spots` + `int top` + `int capacity`） |
| 初始容量 | `TOTAL_SPOTS = 60` |
| 扩容策略 | `top+1 >= capacity` 时 `realloc` 为 2 倍 |
| 操作 | `push`（O(1)）、`pop`（O(1)）、`is_empty`（O(1)）、`size`（O(1)） |
| 用途 | O(1) 车位分配/释放的核心 |

**为什么用栈而不是遍历数组找空位**：遍历 `spots[3][4][5]` 找空闲车位需要 O(n)，而栈的 pop 只需移动栈顶指针，真正 O(1)。这是方案设计中最关键的算法决策。

**内存**：`60 × sizeof(SpotLocation)` ≈ 720 字节（初始），realloc 时翻倍。

### 2.3 哈希表 — `HashMap`

| 属性 | 说明 |
|------|------|
| 类型 | **数组 + 链地址法**（`HTNode *buckets[10007]`） |
| 桶数 | **10007**（质数，减少哈希聚集） |
| 哈希函数 | `hash * 31 + char` 的 Horner 算法，取模 `% HT_SIZE` |
| 冲突解决 | 链表头插法 |
| 存储内容 | `VehicleRecord *`（车牌 → 停放位置 + 入场时间） |
| 操作 | 插入/查找/删除 平均 O(1) |

**关键设计**：

- 键是车牌号字符串，值是 `VehicleRecord*` 指针
- `ht_remove` **只释放 HTNode 节点，不释放 VehicleRecord**（由 `vehicle_exit` 负责 free）
- 哈希表不分页，全部驻留在内存

**内存估算**：桶指针数组 `10007 × 8 bytes ≈ 80KB`，每节点约 40 字节，60 个在场车辆约 2.4KB。

### 2.4 等待队列 — `WaitingQueue`

| 属性 | 说明 |
|------|------|
| 类型 | **单向链表 FIFO**（`QueueNode *front, *rear` + `int count`） |
| 节点内容 | 车牌号 `char[16]`、到达时间 `time_t`、`next` 指针 |
| 操作 | 入队尾 O(1)、出队首 O(1) |

**为什么用链表而不是数组**：队列长度不可预知（理论上可无限排队），动态链表避免预设上限。但缺点是每个节点独立 `malloc`，频繁出入队会产生内存碎片。

### 2.5 历史交易动态数组 — `TransactionRecord *history`

| 属性 | 说明 |
|------|------|
| 类型 | **动态数组**（realloc 扩容） |
| 初始容量 | **100**（在 `ds_init` 中分配） |
| 扩容策略 | `history_count == history_capacity` 时 realloc 扩容 |
| 存储内容 | 车牌、位置、入场/出场时间、停车时长、费用 |
| 用途 | 所有统计功能的数据源（收入统计、楼层热度、峰值时段） |

**问题**：这是一个只增不减的数组，系统运行越久，history 越大。没有归档/清理机制。

### 2.6 顶层聚合结构 — `ParkingSystem`

```c
typedef struct {
    ParkingSpot spots[3][4][5];    // 栈上数组，直接嵌入
    HashMap plate_index;            // 80KB 桶数组嵌入
    FreeSpotStack free_spots;       // 指针+int 嵌入，spots 指向堆
    WaitingQueue wait_queue;        // 两个指针+int 嵌入
    TransactionRecord *history;     // 指针，指向堆
    int history_count, history_capacity;
    FeeRule fee_rule;               // 4 个 double 嵌入
    double total_revenue;
} ParkingSystem;
```

**分配方式**：`ParkingSystem ps;` 在 `main()` 中栈分配。嵌入的结构体（`spots`、`plate_index` 桶数组）也在栈上，内部动态部分（`free_spots.spots`、`history`、队列节点、哈希节点）在堆上。

**重要**：`HashMap` 的 `buckets[10007]` 是 80KB 的指针数组，直接嵌入 `ParkingSystem`。这意味着 `ParkingSystem` 的 `sizeof` 约 80KB+，在 `main()` 的栈上分配。Windows 默认栈大小 1MB，所以没问题，但值得注意。

## 三、持久化存储（二进制文件）

### 3.1 文件格式

```
┌──────────────────────────────────────────────────────┐
│ FileHeader (固定大小)                                 │
│  - magic: 0x504B1204 (文件类型校验)                     │
│  - version: 1                                         │
│  - history_count / free_count / queue_count           │
│  - total_revenue + fee_rule                           │
├──────────────────────────────────────────────────────┤
│ spots[3][4][5] 完整三维数组 (60 × sizeof(ParkingSpot))  │
├──────────────────────────────────────────────────────┤
│ 空闲栈内容 (free_count × sizeof(SpotLocation))          │
├──────────────────────────────────────────────────────┤
│ 等待队列节点 (逐个 fwrite QueueNode)                    │
├──────────────────────────────────────────────────────┤
│ 历史交易 (history_count × sizeof(TransactionRecord))    │
└──────────────────────────────────────────────────────┘
```

### 3.2 各结构持久化情况

| 持久化项 | 是否持久化 | 恢复方式 |
|----------|-----------|---------|
| 车位状态 spots | 是 | 直接 `fread` 覆盖 |
| 空闲栈 | 是 | 读回 `top` 和坐标数组 |
| 等待队列 | 是 | 逐个读 `QueueNode`，调用 `queue_enqueue` 重建 |
| 历史交易 | 是 | 直接 `fread` 到 `malloc` 的数组 |
| 总收入 / 计费规则 | 是 | 从 FileHeader 恢复 |
| **哈希表** | **否** | 加载后遍历 spots 重建 |
| **VehicleRecord.entry_time** | **否（丢失）** | **被替换为 `time(NULL)` 当前时间** |

### 3.3 哈希表不持久化的原因与代价

方案中明确说"哈希表不持久化到文件"，加载后通过扫描 `spots` 三维数组重建。但这里有一个 **数据丢失问题**：

- `spots[f][r][c]` 只存了 `plate` 和 `status`，**不存入场时间**
- 重建哈希表时 `rec->entry_time = time(NULL)` 用了当前时间
- 这意味着：**所有在场车辆的入场时间在重启后全部变成启动时刻**

这是该方案在数据完整性上的最大妥协。如果要修复，有两种办法：

1. 在文件头之后额外写一份 `VehicleRecord` 数组
2. 在 `ParkingSpot` 结构体中增加 `entry_time` 字段

### 3.4 队列节点持久化的隐患

```c
fwrite(p, sizeof(QueueNode), 1, fp);
```

`QueueNode` 包含 `struct QueueNode *next` 指针。`fwrite` 会把这个**内存指针值**写进文件。加载时用 `fread(&node, sizeof(QueueNode), 1, fp)`，读出的 `next` 是上一次运行时的内存地址（无效指针）。不过代码随后用 `queue_enqueue` 重建节点链表，所以 `next` 字段会被正确重设，这个垃圾指针值被无害地忽略了。

## 四、存储方案优缺点总结

| 维度 | 评价 |
|------|------|
| **车位分配** | 空闲栈实现 O(1)，优秀 |
| **车牌查找** | 哈希表实现 O(1)，桶数 10007 对 60 个车位绰绰有余 |
| **排队** | 链表 FIFO，灵活无上限 |
| **统计** | 全部依赖遍历 history 数组，O(n)，数据量小时可接受 |
| **持久化格式** | 二进制格式紧凑，魔数校验防止误读 |
| **入场时间丢失** | 重启后重建哈希表时 entry_time 被重置，**明显缺陷** |
| **history 无限增长** | 无归档/截断机制，长期运行会持续增长 |
| **栈溢出风险** | 80KB+ 的 `ParkingSystem` 在栈上分配，接近极限场景需注意 |
| **跨平台** | 二进制 `fwrite` 结构体存在字节序/对齐/填充差异，不同平台/编译器之间不可移植 |

---

## 五、总体评价

该存储方案在**核心算法性能**（O(1) 车位分配、O(1) 车牌查找）上设计出色，数据结构选型合理（空闲栈 + 哈希表 + 链表队列的组合拳）。但在**数据持久化完整性**上存在入场时间丢失的已知妥协，`ParkingSystem` 栈分配大量内存也值得警惕。整体适合课程实践场景，若用于生产需加固持久化逻辑。
